### Architecture Decisions

#### **1. Templates = Branding Only**
**Decision:** Customer quote templates store ONLY branding (company details + logo + footer)  
**Rationale:**
- Component display rules live in component library (per-component defaults)
- Per-quote overrides in customer quote editor
- Avoids fragile component matching logic
- Simpler UX, less complexity

**Alternative Rejected:** Smart templates with component matching by name

---

#### **2. Quote-First Workflow**
**Decision:** Users customize quotes THEN save as templates  
**Rationale:**
- Natural workflow (work on actual quote, reuse branding later)
- Templates emerge from real work, not abstract planning
- One-click save from editor

**Alternative Rejected:** Template-first creation with empty forms

---

#### **3. Unified Preview Component**
**Decision:** Single `QuotePreview` component used in panel, modal, and view page  
**Rationale:**
- Single source of truth for layout
- Consistency guaranteed (WYSIWYG)
- Easier maintenance

**Alternative Rejected:** Separate preview components per context

---

#### **4. Logo: URL-Based (For Now)**
**Decision:** Logo stored as text URL, user pastes link  
**Rationale:**
- No file upload system needed yet
- Users can host logo anywhere
- Works immediately

**Future Enhancement:** Direct upload to Supabase Storage (Slice H)

### Performance Optimizations

1. **Auto-save Debouncing**
   - 3-second delay after last change
   - Only saves when `isDirty` flag is true
   - Prevents unnecessary database writes

2. **Component Re-rendering**
   - `useCallback` for save handler
   - Stable deps arrays
   - Conditional effects (only run when needed)

3. **Database Efficiency**
   - Batch line saves (single transaction)
   - Delete + insert pattern (clean slate)
   - Indexes on `quote_id` and `sort_order`
