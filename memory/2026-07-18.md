## ROOT CAUSE ANALYSIS — WHY POLYGON OVERSHOOTS IMAGE
**Status: UNRESOLVED — this is the #1 issue to fix in the next session.**

The coordinate mapping math in `computeBackgroundLayout` + `normalizedPointsToCanvas` IS mathematically correct IF:
- `bgImage.width` === natural image width (confirmed: Fabric's `_setWidthHeight` uses `getOriginalSize()` = `element.naturalWidth`)
- `computeBackgroundLayout` uses the same formula as `setPageBackgroundImage` (confirmed: both use `min(800/natW, 600/natH)` and same centering)

**Possible causes still to investigate:**
1. **The AI returns coordinates > 1000.** The validator clamps to 0-1000, but if the AI consistently returns values near 1000 that map to the full image, and the image has white margins, the polygon would correctly cover the image but visually appear to "overshoot" the roof drawing (which is smaller than the image).
2. **The AI is detecting the drawing border/frame or dimension lines as the roof outline**, not the actual roof perimeter. The prompt says "thick black perimeter line" but the AI might interpret the drawing border as the outline.
3. **Image has white margins** — the roof plan image itself has significant white space around the actual roof drawing. The AI correctly identifies coordinates within the image, but when mapped to canvas, the polygon covers the image including margins, which looks like it "overshoots" the roof.
4. **sharp preprocessing changes aspect ratio** — unlikely (`fit: 'inside'` preserves ratio), but should verify by logging the processed image dimensions.
5. **bgImage.width returns something other than naturalWidth** — should verify with console log.

**Next steps for next session:**
- Get the console debug output from Shaun's last test (the `[AI Takeoff] Coordinate mapping debug:` log)
- Check if the AI is returning coordinates that correctly map WITHIN the image bounds but the image has margins
- Consider: should we ask the AI to detect the DRAWING AREA (crop to the roof plan, excluding margins) and then detect the outline within that?
- Consider: add coordinate clamping using `getImageBounds` to prevent any polygon vertex from extending beyond the image bounds on canvas
- The component classification issues (barges vs spouting, ridge lengths) are LIKELY all caused by the same coordinate mapping issue — fix that first, then reassess
